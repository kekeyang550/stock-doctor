# 数据可信度面板设计

日期：2026-07-11

## 背景

当前系统已经有“数据连接器”“数据新鲜度”“刷新记录”三个信息块，但用户需要更直接地判断行情数据是否可信：当前行情来自哪里、是否正在 fallback、缓存是否还能用、最近一次成功刷新是什么时候、刷新任务是否失败。

## 推荐方案

第一阶段先做前端聚合面板，不新增后端接口字段。现有接口已经提供足够信息：

- `/api/v1/system/data-connectors`：活跃 provider、fallback provider、连接器状态、延迟、最近检查时间。
- `/api/v1/system/freshness`：最近成功刷新、刷新年龄、过期阈值、覆盖率、下一步建议。
- `/api/v1/system/refresh-jobs`：最近刷新任务成功/失败、耗时、覆盖数量、消息。

面板在 `DataConnectorPanel` 内升级为“数据可信度”。顶部显示一组紧凑指标：

- 行情来源：`health.active_provider`
- Fallback 状态：若活跃源等于 fallback provider，显示“正在使用回退源”；否则显示“主源直连”
- 缓存状态：根据 `freshness.status` 和 `age_minutes` 显示“缓存可用 / 缓存偏旧 / 缓存过期 / 未建立缓存”
- 最近成功：来自 `freshness.last_success_at`，无值则显示“暂无成功刷新”
- 最近刷新：来自 `jobs[0]`，失败任务明确标红并展示失败消息

## 错误态和空态

本阶段覆盖数据可信度区域的基础空态：

- `health` 未加载：显示“正在检查数据可信度...”
- `freshness` 未加载：缓存指标显示“计算中”
- 无刷新记录：保留“尚未运行刷新任务”
- 最近刷新失败：在摘要区显示“最近刷新失败”，并继续在刷新记录中展示失败原因

更广泛的接口超时、行情失败、无候选股票，会在后续阶段分模块补充，避免一次改动跨太多面板。

## 测试

前端新增 App 级回归测试：

- 加载后能看到“数据可信度”
- 能看到“行情来源 mock”
- 能看到“正在使用回退源”
- 能看到“缓存可用”
- 能看到“最近成功”
- 构造 failed refresh job 时显示“最近刷新失败”

## 不做范围

- 不新增后端接口。
- 不改变 provider 缓存策略。
- 不改刷新任务提交逻辑。
- 不处理 Git 提交和推送；当前本机 Git 提交身份尚未配置。
