# 策略回测样本可信度设计

## 背景

策略回测已经能展示收益、回撤、成本、权益曲线和稳定性评分，但用户还需要判断这些结论是否足够可信。当前页面虽然展示了价格来源、历史样本和 fallback 状态，信息仍偏分散；用户看到回测收益时，不容易快速知道这是历史行情支撑、样本偏少，还是仅由 synthetic trend fallback 得出的样例结果。

## 目标

- 在单策略回测报告中新增样本可信度评分、等级和说明。
- 评分只使用已有回测字段，不新增外部接口或行情请求。
- 前端在“策略回测”区域直观展示可信度，不遮挡核心收益指标。
- HTML 研究报告同步导出可信度信息，离线复盘时保留数据口径。

## 非目标

- 不做真实统计显著性检验。
- 不改变候选股筛选、交易构造、收益计算、周期推荐或策略推荐逻辑。
- 不引入新的第三方依赖。

## 数据模型

`StrategyBacktestReport` 新增字段：

- `sample_confidence_score: int`，范围 0-100。
- `sample_confidence_label: str`，枚举文案为“高 / 中 / 低”。
- `sample_confidence_notes: list[str]`，用于解释样本数量、行情来源、fallback 影响。

前端类型使用可选字段，兼容旧 payload：

- `sample_confidence_score?: number`
- `sample_confidence_label?: string`
- `sample_confidence_notes?: string[]`

## 评分口径

评分由四类已有信息组成：

- 交易样本数：`trade_count >= 8` 得 40 分，`>= 5` 得 30 分，`>= 3` 得 20 分，`> 0` 得 10 分，否则 0 分。
- 行情来源：`historical-kline` 得 35 分，`synthetic-trend` 得 15 分。
- 历史 K 线覆盖：`history_bar_count >= 40` 得 15 分，`> 0` 得 8 分，否则 0 分。
- fallback 状态：未发生 fallback 得 10 分，发生 fallback 得 0 分。

最终分数 clamp 到 0-100。等级映射：

- `>= 75`：高
- `>= 50`：中
- `< 50`：低

## 说明文案

后端生成 3 条说明：

- 样本数量说明，例如“回测交易 6 笔，样本量中等，适合辅助观察。”
- 行情来源说明，例如“使用历史 K 线样本，行情口径更接近真实路径。”
- fallback 说明，例如“发生 fallback：历史行情不足，已切换为样例趋势。”

## 前端展示

- 在“策略回测”的价格来源卡片内新增“样本可信度”和“可信等级”。
- 可信度说明使用跨列说明行展示前 2 条，避免信息过载。
- 旧 payload 缺字段时显示 `0 / 暂无评估`，说明行不渲染。

## 报告导出

HTML 研究报告“策略回测”章节新增：

- 样本可信度
- 可信等级
- 样本可信度说明

所有动态内容继续走 `escapeHtml()`。

## 测试策略

- 后端 focused 测试断言回测报告包含 0-100 的可信度评分、合法等级和非空说明。
- 前端 App 测试断言面板展示“样本可信度 / 可信等级 / 可信度说明”。
- 前端导出测试断言 HTML 包含可信度字段和说明内容。
- 全量验证继续跑后端 pytest、前端 vitest、前端 build、浏览器非白屏和控制台 error 检查。

## 自审

- 无 TBD/TODO。
- 范围聚焦在单策略回测可信度，不触碰筛选和收益算法。
- 字段命名与现有 `stability_*` 风格一致。
- 旧 payload 兼容路径明确。
