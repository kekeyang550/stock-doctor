# 错误态和空态体验设计

日期：2026-07-11

## 背景

当前页面多数接口失败只通过顶部全局错误条提示。诊断主请求失败时，因为 `diagnosis` 为空，主区域会继续显示“正在生成诊断...”，用户无法判断是行情失败、接口超时还是仍在加载。策略股票池和热点选股池已有空态，但文案偏弱，也没有面板级失败反馈。

## 本阶段目标

- 行情/诊断接口失败后，主诊断区显示明确错误态，不再停留在加载中。
- 策略股票池接口失败时，在该面板内显示失败原因和重试按钮。
- 热点选股池接口失败时，在该面板内显示失败原因和重试按钮。
- 无候选股票时，给出更明确下一步建议，而不是只说“暂无”。
- 继续保留顶部全局错误条，用于跨面板提醒；局部面板负责告诉用户哪里出错、能做什么。

## 交互方案

1. 主诊断区：
   - 新增 `diagnosisError`。
   - `loadDiagnosis()` 开始时清空该错误。
   - 失败时保存错误文案，并清空当前诊断相关数据，避免旧数据被误认为新结果。
   - 页面显示“行情数据加载失败”，说明可能是行情源、网络或接口超时，并提供“重试诊断”按钮。

2. 策略股票池：
   - 新增 `screenerError`。
   - 接口成功时清空错误并更新候选。
   - 接口失败时保留当前候选为空，并在面板显示“策略股票池加载失败”和“重试”按钮。
   - 空态改为“当前策略没有候选股票”，并提示可切换策略或等待下一轮行情刷新。

3. 热点选股池：
   - 新增 `hotspotCandidatesError`。
   - 模式切换或加载失败时显示面板级错误与“重试”按钮。
   - 空态改为“当前热点模式没有候选股票”，并提示可切换到均衡/激进模式或等待热点刷新。

## 非目标

- 不新增后端 API。
- 不引入 toast 系统。
- 不做全站错误边界。
- 不改变现有数据源 fallback 和缓存逻辑。

## 验收

- 新增前端测试覆盖诊断失败、策略池失败、热点池失败、无候选空态。
- 前端全量测试、构建和后端测试继续通过。
